## Background: Autocomplete as a Search Problem

Alright! Let's give you some context before you get into the weeds of the starter code. 
Autocomplete might seem like some complicated magic, but at its core, it's just an application of search algorithms on a tree (that's how it's done in this assignment for your simplicity, but it's done very differently in real word). Let's break down how this works:

**The Search Space: A Tree of Characters**

To implement the autocomplete feature, you would build a tree of characters, which will be the search space for this search problem. 
In your starter code, you're given a `document` (a `txt` file) of several words. 
Imagine each word in your document is broken down into its individual letters. Now, picture these letters arranged in a single tree-like structure, for example look at the tree diagram below:

- This is what the search space for your search problem would look like. 
- You will traverse the *tree* starting from the last node of the prefix that the user enters to generate autocomplete suggestions. 

**The Search Problem**

When a user types a prefix (e.g., "ca"), the autocomplete feature needs to find all the words in the *tree* that start with that prefix. This translates to a search problem:

- **Initial state:** The node representing the last letter of the prefix ("a" in our example).
- **Action** - a transition between one letter to the next letter in the *tree*
- **Goal:** The end of the word(s) (that start with the given prefix) in the *tree*. <u>Note how there could be multiple goals in this problem.</u>
- **Path:** The sequence of characters from the root to a goal node represents a complete word.

**Search Algorithms**

We can employ various search algorithms to traverse this *tree* and find our goal nodes (complete words).

- **Breadth-First Search (BFS):**  Explores the *tree* level-by-level, ensuring we find the shortest words first. 
- **Depth-First Search (DFS):** Dives deep into the *tree*, potentially finding longer, less common words first.
- **Uniform-Cost Search (UCS):** Considers the frequency of each character transition to prioritize more likely words based on the prefix.

**Multiple Goals and Paths**

In autocomplete, we're not just looking for a single goal node. We want to find *all* the goal nodes (words) that follow from the prefix. Furthermore, we're interested in the entire path from the root to each goal node, as this path represents the complete suggested word.

**Your Task:**

Your task is to implement BFS, DFS, and UCS to traverse the *tree* and generate autocomplete suggestions. You'll see how different algorithms affect the order and type of words suggested, and understand the trade-offs involved in choosing one over the other.

### Recursive DFS vs Stack-based DFS
- Explain your intuition in recursive DFS VS stack-based DFS, and which one you used here.

I chose a stack-based DFS for this problem, and my reasoning resides in the difference between the two approaches, with respect to this problem. 

In a recursive DFS, the "leftmost" children of a certian node are visited first, whereas in a stack-based DFS, the "rightmost" children are visited first. In this context, the leftmost child of a node is the first word found in the txt file with a certain prefix, and the rightmost child of a node is the last node found in the txt file with a certain prefix. While it wasn't made clear what the document files containing the words represented, I presumed that words were ordered by the recency of their use in some collection of digital messages, with the most recently used word being the last word in the document. 

With this assumption, since chars inserted into a dictionary as keys are ordered by when they were inserted, the rightmost child of a certain node represents the most recently used word stemming from that child. Since stack-based DFS visited rightmost nodes first, this method will result in recommendations that prioritize words that have been used more recently. 

In my opinion, this makes sense for an autocomplete algorithm. If I typed the word "thaw" once, than proceed to type "that" twenty straight times, I shouldn't get the word "thaw" recommended first because it was the first word that was typed. The stack-based approach allows the top recommendations to get somewhat updated as new words are inputted.

## Experimental
- Explain here what differences did you see in the suggestions generated when you used BFS vs DFS vs UCS. 

Here is a summary of the observations I've made when exploring the differences between BFS, DFS, and UCS algorithms:
    - BFS will recommend words in order of length
        - Words with shorter length will be recommended earlier
        - Within each word length, words are recommended based on the order in which they were inserted into the tree, with words entered earlier being more highly recommended 
    - BFS can do well at finding short words close in letter count to the prefix that we've already typed out, but this doesn't account for the recency we've typed these words, or the frequency at which we've typed them

    - DFS will, at each step, recommend the word that was most recently entered into the tree
        - For instance, when searching with the prefix 'be', bestie is recommended before bet since bestie was the most recently entered word
        - Thus, DFS tells us to follow the 's' branch instead of the 't' branch
    - DFS can do well at recommending recent words, but it doesn't consider how often the words are used
        - For example, if the word 'yak' was entered 50 times and then we entered 'yankee', this DFS algorithm would recommend yankee when its probably more reasonable to recommend             yak

    - UCS will create a cost algorithm that strives to match choices with costs, where choosing a lower costs branch indicates choosing a 'more likely' branch(the number of times           we've seen words on this branch exceeds the number of times we've seen words on the other branch)
        - UCS will recommend words almost always in order of frequency
        - The top recommendations will be the words that occur most often
        - This allows us to get recommendations that are weighted based on 'generally' how likely we are to type them out
     - UCS does great at recommending common words, but its ranking fails to consider how recently these words were typed
        - If we accidentally type the word 'bob' one thousand times by copy-pasting it over and over, the #1 recommended word before we start typing any single word will be 'bob'
